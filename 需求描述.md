# 📜 Laplace 项目工程简报 (Project Truth)

## 一、 项目背景与初衷 (Project Context)
*   **起源**: 本项目旨在重构 FGO（Fate/Grand Order）玩家的工具体验。传统的工具软件（如 **Chaldea**）虽然功能强大，但依赖于复杂的图形用户界面（GUI）和多重筛选逻辑。
*   **愿景**: 利用大语言模型（LLM）的意图识别能力，将工具转化为基于自然语言（LUI）的智能助手 —— **Laplace**。
*   **产品定位**: **AI Native** 对话式 FGO 数据助手。用户通过自然语言与 Laplace 对话，无需学习任何筛选 UI。
*   **进化**: 
    *   **Old Way**: 打开 App -> 选择从者列表 -> 点击筛选 -> 勾选 NP 充能 -> 输入 30。
    *   **New Way (Laplace)**: 用户在对话框输入："帮我找一下 30 自充的从者有哪些。" -> LLM 解析意图 -> 生成查询指令 -> 执行数据筛选 -> 对话框直接返回渲染好的从者卡片。
*   **核心理念**: **Vibe Coding**。通过自然语言驱动代码生产，强调开发者与 AI 之间的逻辑共鸣，而非繁琐的手动编码。

## 二、 核心架构：AI Native 对话式查询

### 2.1 系统流程

```
用户自然语言输入
       ↓
┌─────────────────┐
│   LLM 意图解析   │  ← 将自然语言转换为结构化查询指令
│  (Intent Parse)  │
└────────┬────────┘
         ↓
┌─────────────────┐
│  查询指令执行器   │  ← 基于指令查询 FGO 数据库
│ (Query Executor) │
└────────┬────────┘
         ↓
┌─────────────────┐
│  LLM 结果格式化  │  ← 将结果组织成自然语言 + 卡片
│ (Response Gen)   │
└────────┬────────┘
         ↓
    对话框返回结果
```

### 2.2 意图解析示例

| 用户输入 | LLM 解析结果 |
| :--- | :--- |
| "30 自充的从者有哪些" | `{"action": "filter", "target": "servant", "condition": {"npCharge": {"eq": 30}, "targetType": "self"}}` |
| "大于 50 自充的从者" | `{"action": "filter", "target": "servant", "condition": {"npCharge": {"gte": 50}, "targetType": "self"}}` |
| "有哪些五星 Caster 有自充" | `{"action": "filter", "target": "servant", "condition": {"rarity": 5, "class": "caster", "hasNpCharge": true}}` |
| "呆毛的技能有什么效果" | `{"action": "detail", "target": "servant", "identifier": "呆毛"}` |

### 2.3 关键设计决策

1.  **LLM 不直接查数据**: LLM 只负责"意图解析"和"结果格式化"，不直接操作数据。
2.  **中间层执行**: 解析出的结构化 JSON 指令由 Query Executor 执行，确保可预测性。
3.  **Strict JSON Schema**: LLM 的输出必须符合预定义的 JSON Schema，确保可靠性。

## 四、 深度分析指令 (Analysis Directives for Agent)
**Agent 启动后应优先对挂载的 `chaldea` 源码执行以下分析：**

1.  **数据模型审计 (Data Models)**:
    *   分析 `lib/generated/models/gamedata/servant.g.dart` 和 `skill.g.dart`。
    *   **重点**: 识别从者技能数据在 JSON 中的嵌套层级。理解 `functions` 数组如何存储效果（如 `gainNp`）以及 `svals` 如何存储不同等级的数值。

2.  **核心逻辑映射 (Core Logic)**:
    *   分析 `lib/app/battle/functions/` 下的各 Dart 类文件。
    *   **重点**: 提取 `gain_np.dart` 的计算逻辑，用于 Query Executor 准确计算从者自充。

3.  **别名系统研究 (Alias & Metadata)**:
    *   分析 `res/assets/` 目录下的 JSON 资产。
    *   **重点**: 找到从者别名字典，确保 AI 能理解"呆毛"、"村正"、"C呆"等非规范术语。

## 五、 架构选型与哲学 (Engineering Principles)

1.  **Web 优先，渐进增强**:
    *   **MVP 阶段**: Web 单页应用（HTML/CSS/JS），内嵌对话框 UI。
    *   **后续阶段**: 使用 **SwiftUI** + **SwiftData** 构建 iOS 原生界面。
    *   **逻辑层**: Python 后端服务，集成 LLM API，暴露 RESTful 接口。

2.  **协议约束 (Strict Protocol)**:
    *   **JSON Only**: 严格拒绝 YAML。所有配置、Prompt、端云数据交换必须遵循 **JSON Schema**。
    *   **理由**: 确保 AI 生成的数据流具有可预测性和结构稳定性。

3.  **合规性设计 (Compliance)**:
    *   **协议**: 遵循 **CC-BY-NC-SA-4.0** (非商业性使用、相同方式共享)。
    *   **声明**: 在项目显著位置注明："数据及部分逻辑源自开源项目 Chaldea"。

## 七、 下一步任务：AI Native 对话式查询

### 7.1 MVP 目标
在 Web Demo 中添加**对话框界面**，用户输入自然语言问题，通过 LLM 解析并执行查询，在对话框中返回结果。

### 7.2 具体交互流程
1.  用户在底部对话框输入："30NP 自充的从者有哪些？"
2.  前端将问题发送至后端 API
3.  后端调用 LLM（如 OpenAI / Gemini / Claude）解析意图
4.  LLM 返回结构化查询指令（JSON）
5.  Query Executor 执行查询，从预处理的从者数据中筛选
6.  结果通过 LLM 格式化后返回前端
7.  前端在对话框中渲染从者卡片 + 文字说明

### 7.3 技术选型
| 组件 | 技术 | 说明 |
| :--- | :--- | :--- |
| 前端 | HTML/CSS/JS | 对话框 UI + 从者卡片渲染 |
| 后端 | Python (FastAPI) | API 服务 + LLM 集成 |
| LLM | OpenAI / Gemini / Claude API | 意图解析 + 结果格式化 |
| 数据 | 预处理 JSON | 从 Atlas Academy API 导出 |
