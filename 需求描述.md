# 📜 Laplace 项目工程简报 (Project Truth)

## 一、 项目背景与初衷 (Project Context)
*   **起源**: 本项目旨在重构 FGO（Fate/Grand Order）玩家的工具体验。传统的工具软件（如 **Chaldea**）虽然功能强大，但依赖于复杂的图形用户界面（GUI）和多重筛选逻辑。
*   **愿景**: 利用大语言模型（LLM）的意图识别能力，构建一个**功能覆盖度对标 Chaldea**、但以自然语言（LUI）为唯一交互方式的智能助手 —— **Laplace**。
*   **产品定位**: **AI Native** 对话式 FGO 全数据助手。用户通过自然语言与 Laplace 对话，获得与 Chaldea 同等精度和广度的数据查询能力，无需学习任何筛选 UI。
*   **进化**:
    *   **Old Way**: 打开 App -> 选择从者列表 -> 点击筛选 -> 勾选 NP 充能 -> 输入 30。
    *   **New Way (Laplace)**: 用户在对话框输入：「帮我找一下 30 自充的从者有哪些。」 -> LLM 解析意图 -> 生成查询指令 -> 执行数据筛选 -> 对话框直接返回渲染好的从者卡片。
*   **核心理念**: **Vibe Coding**。通过自然语言驱动代码生产，强调开发者与 AI 之间的逻辑共鸣，而非繁琐的手动编码。

## 二、 核心架构：AI Native 对话式查询

### 2.1 系统流程

```
用户自然语言输入
       ↓
┌─────────────────┐
│   LLM 意图解析   │  ← 将自然语言转换为结构化查询指令
│  (Intent Parse)  │     基于 Schema Mirror 知识库理解 FGO 领域语义
└────────┬────────┘
         ↓
┌─────────────────┐
│  查询指令执行器   │  ← 基于指令查询 FGO 全效果数据库
│ (Query Executor) │
└────────┬────────┘
         ↓
┌─────────────────┐
│  LLM 结果格式化  │  ← 将结果组织成自然语言 + 卡片
│ (Response Gen)   │
└────────┬────────┘
         ↓
    对话框返回结果
```

### 2.2 意图解析示例

| 用户输入 | 查询类型 | LLM 解析结果 |
| :--- | :--- | :--- |
| 「30 自充的从者有哪些」 | NP 充能 | `{npCharge: {eq: 30}}` |
| 「五星 Caster 有自充的从者」 | NP 充能 + 职阶 + 稀有度 | `{npCharge: {gte: 1}, rarity: {eq: 5}, className: "caster"}` |
| 「有无敌技能的从者」 | 效果筛选 | `{skillEffect: "invincible"}` |
| 「可以给全队加攻的 Saber」 | 效果 + 目标 + 职阶 | `{skillEffect: "upAtk", targetType: "party", className: "saber"}` |
| 「有回避且带毅力的从者」 | 多效果组合 | `{skillEffects: ["avoidance", "guts"]}` |
| 「呆毛的技能分别是什么效果」 | 从者详情 | `{action: "detail", identifier: "呆毛"}` |
| 「有即死技能的 Assassin」 | 效果 + 职阶 | `{skillEffect: "instantDeath", className: "assassin"}` |

### 2.3 关键设计决策

1.  **LLM 不直接查数据**: LLM 只负责"意图解析"和"结果格式化"，不直接操作数据。
2.  **中间层执行**: 解析出的结构化 JSON 指令由 Query Executor 执行，确保可预测性。
3.  **Strict JSON Schema**: LLM 的输出必须符合预定义的 JSON Schema，确保可靠性。
4.  **Schema Mirror 策略**: 不直接引用 Chaldea 的 Dart 代码，而是提取其**领域知识**（效果分类、枚举映射、数据路径）作为 LLM 的知识库。

## 三、 知识库架构：Schema Mirror

### 3.1 设计理念

Chaldea 项目用 Dart 构建了 FGO 最完整的数据类型系统（165+ 种 FuncType、200+ 种 BuffType）。Laplace 不复制其代码，而是**提取其领域建模知识**，注入到 LLM 的 System Prompt 中。

### 3.2 知识提取层次

```
Chaldea Dart 源码
       ↓  提取
┌─────────────────────────────────────┐
│  Layer 1: SkillEffect 效果分类体系   │  40+ 种人类可理解的效果分类
│  Layer 2: FuncType / BuffType 枚举   │  底层效果类型完整列表
│  Layer 3: MappingData 多语言映射     │  中日英韩从者名/技能名/职阶名
│  Layer 4: 数据路径约定               │  skills→functions→svals 的嵌套结构
└─────────────────────────────────────┘
       ↓  注入
┌─────────────────────────────────────┐
│  server/knowledge/                   │
│  ├── effect_schema.json              │  效果分类 → FuncType/BuffType 映射
│  ├── class_mapping.json              │  职阶中日英映射
│  ├── func_types.json                 │  FuncType 枚举完整列表
│  ├── buff_types.json                 │  BuffType 枚举完整列表
│  └── alias_mapping.json             │  从者别名字典
└─────────────────────────────────────┘
       ↓  融入
   LLM System Prompt
```

### 4.1 知识库同步策略

当 Chaldea 的领域建模知识更新（新从者、新 FuncType、新 BuffType 等）时，Laplace 通过 **`sync_chaldea.py` 同步脚本** 一键完成知识更新：

```
Chaldea Dart 源码更新 (git pull)
              ↓
    python3 server/sync_chaldea.py
              ↓
┌──────────────────────────────────────────────┐
│  Step 1: 解析 func.dart    → FuncType 枚举    │
│  Step 2: 解析 buff.dart    → BuffType 枚举    │
│  Step 3: 解析 effect.dart  → SkillEffect 映射 │
│  Step 4: 解析 common.dart  → SvtClass 枚举   │
│  Step 5: 解析 mappings.dart → 多语言别名      │
└──────────────────┬───────────────────────────┘
                   ↓  自动输出
┌──────────────────────────────────────────────┐
│  server/knowledge/                            │
│  ├── func_types.json     ← FuncType 完整列表  │
│  ├── buff_types.json     ← BuffType 完整列表  │
│  ├── effect_schema.json  ← 效果分类映射       │
│  ├── class_mapping.json  ← 职阶枚举          │
│  └── _meta.json          ← 版本号 + 时间戳    │
└──────────────────┬───────────────────────────┘
                   ↓  触发
          data_loader.py 刷新从者数据
                   ↓
          System Prompt 自动加载最新知识
```

**设计原则**：

1. **纯正则解析** — 不依赖 Dart SDK，Python 正则直接从 `.dart` 源文件中提取枚举值和映射关系
2. **版本追踪** — `_meta.json` 记录 Chaldea 的 git commit hash + 提取时间戳，方便判断是否需要更新
3. **幂等操作** — 重复运行不会产生副作用，每次输出覆盖旧文件
4. **单命令同步** — `python3 server/sync_chaldea.py` 一条命令完成全部提取
5. **CI 友好** — 可挂到 GitHub Actions 定时执行（如每周自动 pull Chaldea + 重新提取 + 提交知识库变更）

**完整更新流程**（手动）：

```bash
# 1. 更新 Chaldea 源码
cd chaldea-center/chaldea && git pull origin main && cd ../..

# 2. 重新提取知识库
python3 server/sync_chaldea.py

# 3. 刷新从者数据库（如果 Atlas API 有新数据）
python3 server/data_loader.py

# 4. 重启后端服务（自动加载新知识）
python3 -m uvicorn server.main:app --reload
```

## 五、 深度分析指令 (Analysis Directives for Agent)
**Agent 启动后应优先对挂载的 `chaldea` 源码执行以下分析：**

1.  **效果分类映射 (SkillEffect)**:
    *   分析 `lib/models/gamedata/effect.dart`。
    *   **重点**: 提取 40+ 种效果分类的 FuncType/BuffType 映射关系，生成 `effect_schema.json`。

2.  **枚举完整列表 (FuncType / BuffType)**:
    *   分析 `lib/models/gamedata/func.dart` 和 `buff.dart`。
    *   **重点**: 提取所有枚举值及其 int 编号，生成 `func_types.json` 和 `buff_types.json`。

3.  **数据模型审计 (Servant / Skill / Function / DataVals)**:
    *   分析 servant.dart、skill.dart、func.dart、vals.dart。
    *   **重点**: 理解 `skills[] → functions[] → svals[9].Value` 的数据路径。

4.  **多语言映射 (MappingData)**:
    *   分析 `lib/models/gamedata/mappings.dart`。
    *   **重点**: 提取 svtNames / skillNames / className 的中日英映射。

5.  **别名系统研究 (Alias & Metadata)**:
    *   分析 `res/assets/` 和 MappingData。
    *   **重点**: 构建从者别名字典（「呆毛」→ Artoria、「C呆」→ Altria Caster、「村正」→ Muramasa）。

## 六、 架构选型与哲学 (Engineering Principles)

1.  **Web 优先，渐进增强**:
    *   **MVP 阶段**: Web 单页应用（HTML/CSS/JS），内嵌对话框 UI。
    *   **后续阶段**: 使用 **SwiftUI** + **SwiftData** 构建 iOS 原生界面。
    *   **逻辑层**: Python 后端服务，集成 LLM API，暴露 RESTful 接口。

2.  **协议约束 (Strict Protocol)**:
    *   **JSON Only**: 严格拒绝 YAML。所有配置、Prompt、端云数据交换必须遵循 **JSON Schema**。
    *   **理由**: 确保 AI 生成的数据流具有可预测性和结构稳定性。

3.  **合规性设计 (Compliance)**:
    *   **协议**: 遵循 **CC-BY-NC-SA-4.0** (非商业性使用、相同方式共享)。
    *   **声明**: 在项目显著位置注明：「数据及部分逻辑源自开源项目 Chaldea」。
