# 架构讨论记录 (Architecture Decision Records)

本目录用于记录 Laplace 项目的架构设计讨论、方案对比和决策过程。

## 用途

在项目开发过程中，经常会遇到需要权衡不同技术方案的场景。本目录用于：

1. **记录讨论过程**：保存不同方案的优缺点对比
2. **记录决策依据**：说明为什么选择某个方案而非其他
3. **保留未来参考**：当方案需要调整时，可以回溯历史讨论
4. **知识沉淀**：为后续开发者提供架构演进的全景图

## 文件命名规范

```
ADR-XXX-主题.md
```

- `ADR`: Architecture Discussion Record
- `XXX`: 序号（如 013）
- `主题`: 简短描述（如 skill-architecture）

## 文档结构

每个 ADR 文档应包含：

```markdown
# ADR-XXX: 主题

**日期**: YYYY-MM-DD  
**状态**: 讨论中 (Discussing) / 已决策 (Decided) / 已废弃 (Deprecated)  
**参与者**: ...  
**相关 Issue**: ...  

---

## 一、问题背景
- 描述需要解决的痛点
- 相关日志/数据分析

## 二、方案对比
- 方案 A：优势、劣势、成本
- 方案 B：优势、劣势、成本
- 推荐方案及理由

## 三、实施计划
- 分阶段实施步骤
- 工作量评估
- 风险评估

## 四、关键决策记录
- 决策 1：选择 X 而非 Y，原因...
- 决策 2：...

## 五、后续行动
- [ ] 待办事项 1
- [ ] 待办事项 2

## 六、参考资料
- 相关文档链接
- 技术文档链接
```

## 工作流程

1. **发起讨论**：创建新的 ADR 文档，记录问题和方案
2. **讨论中**：在文档中补充各方意见、补充方案
3. **达成决策**：更新状态为"已决策"，记录最终方案
4. **实施**：将方案细节更新到 `需求描述.md` 等实施文档
5. **归档**：实施完成后，更新状态为"已归档"

## 与实施文档的关系

```
架构讨论 (docs/architecture-discussions/)
    ↓ 方案明确后
实施计划 (需求描述.md)
    ↓ 开始编码
代码实现 (server/*.py)
    ↓ 完成
文档更新 (README.md, MEMORY.md)
```

**原则**：
- 架构讨论文档保留完整的思考过程
- 实施文档（需求描述.md）只记录最终决策和实施步骤
- 避免在实施文档中保留大量已废弃的方案
